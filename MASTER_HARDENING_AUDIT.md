## Priority 1 — Document Template System

### 1A. Architectural Consolidation

The system currently has **5 PDF libraries** and **4 independent cover page implementations**. This is the single biggest source of branding/rendering inconsistency.

#### Current State

| Cover Page System                             | Library             | Used By                                       | Branding Level |
| --------------------------------------------- | ------------------- | --------------------------------------------- | -------------- |
| `src/lib/pdf/shared/pdfHeader.ts`             | jsPDF (server)      | Weather report, template PDF, plan export     | ✅ Full        |
| `src/components/pdf/CoverPageSection.tsx`     | jsPDF (client)      | Depreciation invoice, material estimator      | ✅ Full        |
| `src/components/reports/shared/CoverPage.tsx` | @react-pdf/renderer | Universal claims report, rebuttal, supplement | ✅ Full        |
| `src/lib/reports/core/sections/CoverPage.ts`  | pdf-lib             | Module reports orchestrator                   | ✅ Full        |
| `src/lib/pdf/claimPacketPdf.ts` (inline)      | pdf-lib             | Full claim packet                             | ❌ None        |

| PDF Library         | Files Using | Notes                                         |
| ------------------- | ----------- | --------------------------------------------- |
| jsPDF               | ~12 files   | Primary for weather, templates, plan exports  |
| @react-pdf/renderer | ~8 files    | Claims reports, rebuttals, supplements        |
| pdf-lib             | ~5 files    | Claim packet, module reports, financial audit |
| PDFKit              | ~3 files    | Damage report, master report, AI report       |
| Puppeteer           | ~2 files    | ⚠️ Will fail on Vercel serverless             |

#### Audit Checklist — Cover Page Structure

Every generated PDF cover page MUST follow this locked structure:

**Top Header:**

- [ ] 🔴 Report type title (e.g., "Weather & Loss Justification Report")
- [ ] 🔴 Branding row: company logo (left) · company name + user name + contact info (center) · employee headshot (right)
- [ ] 🟡 Fallback: colored circle with company initial when no logo loads
- [ ] 🟡 Fallback: no headshot → graceful omission (no broken image)

**Middle Hero Section:**

- [ ] 🔴 Property image or map visual
- [ ] 🔴 Client name above property address
- [ ] 🔴 Property address below client name
- [ ] 🟡 Claim/project type below address

**Lower Metadata Section (Insurance):**

- [ ] 🔴 Insured name
- [ ] 🔴 Carrier name
- [ ] 🔴 Policy number
- [ ] 🔴 Claim number
- [ ] 🔴 Date of loss
- [ ] 🔴 Report generated date

**Lower Metadata Section (Retail):**

- [ ] 🔴 Client name
- [ ] 🔴 Project type
- [ ] 🔴 Scope type
- [ ] 🟡 Project status
- [ ] 🟡 Estimate date
- [ ] 🟡 Assigned rep

**Footer:**

- [ ] 🟡 "Presented by [Company Name]"
- [ ] 🟡 "Generated by SkaiScraper"
- [ ] 🟡 Page numbers

## Priority 3 — Estimator Completeness Audit

### 3A. Roof Pitch Options

**Current state:** Material Estimator defines pitches inline in the page component (not shared):

```
2/12, 3/12, 4/12, 5/12, 6/12, 7/12, 8/12, 9/12, 10/12, 11/12, 12/12
```

**Missing:** `0/12` (dead flat) and `1/12` (low-slope commercial).

**Worse:** Three different pitch format systems exist:

| Location                         | Format               | Values                                                 |
| -------------------------------- | -------------------- | ------------------------------------------------------ |
| Material Estimator page          | Exact ratio `"6/12"` | 2/12 – 12/12                                           |
| Step4_RoofDetails (claim intake) | Free text input      | User types anything                                    |
| Inspection Overview              | Category buckets     | `flat`, `low-slope`, `moderate`, `steep`, `very-steep` |

- [ ] 🔴 **Create shared pitch constant** in `src/lib/constants/roofing.ts` — all three locations should reference it
- [ ] 🔴 **Add 0/12 and 1/12** to the pitch options list
- [ ] 🟡 **Standardize pitch format** — pick one format (`"6/12"` ratio is most industry-standard) and convert category buckets to it
- [ ] 🟡 Consider adding steep-slope options: `14/12`, `16/12`, `18/12` for mansard/A-frame

### 3B. Roofing Material Options

**Current:** Material Estimator only supports 3 asphalt shingle types:

- 3-Tab Asphalt
- Architectural Shingle
- Designer / Premium

**Missing from estimator (but exist elsewhere in the codebase):**

- Metal
- Tile
- Slate
- TPO
- EPDM
- Wood Shake

Step4_RoofDetails has: `asphalt-3tab`, `asphalt-architectural`, `metal`, `tile`, `slate`, `wood`, `tpo`, `epdm`
Inspection Overview has: `asphalt-shingle`, `metal`, `tile`, `flat`, `wood-shake`, `slate`

**⚠️ These two lists use different value formats** (e.g., `wood` vs `wood-shake`).

- [ ] 🔴 **Add metal, tile, TPO, EPDM, wood shake, slate** to the Material Estimator (even if AI-estimated rather than deterministic)
- [ ] 🔴 **Create shared material constants** — unify `wood` vs `wood-shake` vs `wood_shake`
- [ ] 🟡 **Extend deterministic engine** beyond asphalt shingles (at least for metal and tile)

## Priority 6 — Permissions & Role Audit

### 6A. Critical Finding: Three Parallel RBAC Systems

⚠️ This is the biggest permissions risk. Three independent systems exist:

| System   | File                          | Roles                                                          | DB Table                            | Permissions                   |
| -------- | ----------------------------- | -------------------------------------------------------------- | ----------------------------------- | ----------------------------- |
| System A | `src/lib/rbac.ts`             | OWNER, ADMIN, MANAGER, FIELD_TECH, SALES_REP, VIEWER (6 roles) | `users.role` + `user_organizations` | 17 permissions                |
| System B | `src/lib/auth/rbac.ts`        | admin, manager, member, viewer (4 roles)                       | `team_members` → `users`            | 23 permissions                |
| System C | `src/lib/auth/permissions.ts` | ADMIN, MANAGER, MEMBER (3 roles)                               | `user_organizations`                | 9 resources × 5 action levels |

The client-side `RBACGuard` mirrors **System A**. Server-side API routes may use **System B** or **System C**. A user could be `ADMIN` in one system and have no entry in the other.

- [ ] 🔴 **Pick ONE canonical RBAC system** — recommended: consolidate on System B (`src/lib/auth/rbac.ts`) as it has the most permissions and HOF wrappers
- [ ] 🔴 **Update client-side RBACGuard** to match the chosen system's role names
- [ ] 🔴 **Retire deprecated systems** — mark Systems A and C as deprecated, create migration plan
- [ ] 🔴 **Remove hardcoded platform owner email** in System B (`buildingwithdamienray@gmail.com`) → use `PLATFORM_OWNER_EMAIL` env var
- [ ] 🔴 **Remove emergency auto-ADMIN on DB failure** in System B — this could grant admin access during transient errors
